# Tampermonkey API Recorder 方案拆解

## 1. 项目定义

### 1.1 一句话定义

一个面向测试工程师的油猴脚本。在网页正常操作过程中自动捕获接口请求，判断成功或失败，沉淀可回放的接口模板，并输出测试报告。

### 1.2 目标用户

- 日常做功能测试、回归测试的测试工程师
- 需要快速收集接口资产但不想手工抓包的人
- 需要在页面操作过程中同步发现接口失败的人

### 1.3 要解决的问题

- 页面操作时接口很多，手工记录成本高
- 测试过程中接口是否成功通常要额外打开 Network 面板确认
- 后续回归时缺少可复用的接口模板
- 失败接口无法及时沉淀成结构化报告

## 2. 产品边界

### 2.1 MVP 范围

第一版只做下面这些能力：

- 自动捕获 `fetch` / `XMLHttpRequest`
- 对命中的接口做成功 / 失败 / 未知判定
- 失败时实时弹出提示
- 成功请求保存到本地
- 提供悬浮面板查看记录
- 支持单接口回放
- 支持批量回放已保存的成功接口
- 导出 JSON / HTML 报告

### 2.2 明确不做

第一版先不支持以下复杂场景：

- 文件上传下载
- WebSocket / SSE
- 非同源 iframe 深度接管
- 强依赖签名算法的接口自动修复
- 复杂场景编排回放
- 团队共享存储
- 自动登录

### 2.3 使用前提

- 默认仅在测试环境开启回放能力
- 页面本身能正常登录
- 用户明确知道回放可能产生数据副作用

## 3. 核心用户流程

### 3.1 录制流程

1. 用户打开页面，脚本注入
2. 脚本劫持 `fetch` 和 `XMLHttpRequest`
3. 用户正常操作页面
4. 命中的请求被记录
5. 响应返回后，脚本做状态判定
6. 成功请求入库，失败请求提示并入库

### 3.2 回放流程

1. 用户打开悬浮面板
2. 选择一条成功记录或一个接口集合
3. 脚本生成待回放请求
4. 动态替换鉴权变量
5. 发起请求并收集结果
6. 生成回放报告

### 3.3 报告流程

1. 用户结束本轮测试
2. 进入报告页签
3. 查看统计汇总和失败详情
4. 导出 JSON 或 HTML

## 4. 系统设计

### 4.1 模块划分

建议按下面 8 个模块拆：

- `capture`：拦截和采集请求
- `normalize`：统一格式化请求/响应
- `classifier`：成功失败判定
- `storage`：本地存储
- `replay`：变量替换和回放执行
- `report`：统计和导出
- `ui`：悬浮按钮、面板、toast
- `config`：接口匹配规则、环境开关、字段脱敏规则

### 4.2 运行结构

油猴脚本内部可以拆成三层：

- 注入层：负责改写 `window.fetch` 和 `XMLHttpRequest`
- 业务层：负责判定、存储、回放、报告
- 展示层：负责悬浮 UI 和用户交互

### 4.3 数据流

1. 浏览器请求发出
2. `capture` 拿到请求元数据
3. 请求响应完成后进入 `normalize`
4. `classifier` 产出 `success` / `fail` / `unknown`
5. `storage` 持久化
6. `ui` 刷新统计
7. 用户触发 `replay`
8. `report` 汇总结果导出

## 5. 存储方案

### 5.1 结论

第一版推荐：

- `IndexedDB`：存接口记录、回放模板、报告
- `GM_setValue`：存轻量配置和用户偏好
- 导出 `JSON`：做备份、迁移、分享

### 5.2 为什么不用文件作为主存储

- 油猴对本地文件持续写入不友好
- 文件查询、去重、筛选能力差
- 数据一多后不利于增量更新
- 回放和报告都需要结构化查询能力

### 5.3 数据分仓

建议建立 4 张逻辑表：

- `sessions`：一次测试会话
- `records`：录制到的接口记录
- `templates`：可回放接口模板
- `replay_reports`：回放结果报告

## 7. 成功 / 失败判定设计

### 7.1 三段式判定

不要只看 HTTP 状态码。建议三层组合：

- HTTP 层：`2xx` 视为候选成功
- 业务层：检查 `code`、`success`、`msg` 等业务字段
- 规则层：按接口路径覆盖默认规则

### 7.2 判定结果

统一产出 3 种状态：

- `success`
- `fail`
- `unknown`

`unknown` 很关键，避免误判。

### 7.3 判定规则示例

```json
[
  {
    "match": "POST /api/order/create",
    "successWhen": [
      "status in 200..299",
      "body.code === 0"
    ],
    "failWhen": [
      "body.code !== 0"
    ]
  }
]
```

### 7.4 默认策略

- 响应不是 JSON 时，优先标记为 `unknown`
- HTTP 非 `2xx` 时，默认标记为 `fail`
- JSON 中存在常见失败字段时，优先标记为 `fail`

## 8. 回放设计

### 8.1 核心原则

回放保存的不是“原始请求”，而是“请求模板 + 动态变量规则”。

### 8.2 鉴权替换策略

支持以下变量来源：

- `cookie`
- `localStorage`
- `sessionStorage`
- `用户手动输入`

### 8.3 回放模式

- 单条回放：验证某一个接口是否还能成功
- 批量回放：一次回放当前已保存的成功接口
- 仅当前域名回放：避免跨环境误发

### 8.4 跳过规则

下列情况直接跳过并记录原因：

- 缺少必须变量
- 请求命中黑名单
- 请求被标记为高风险副作用
- 环境不匹配

### 8.5 风险控制

模板上增加风险标签：

- `safe`：读操作，可直接回放
- `warning`：可能写数据，回放前确认
- `danger`：明确有副作用，默认禁用

## 9. UI 方案

### 9.1 入口

页面右下角固定一个悬浮按钮，显示当前会话统计：

- 成功数
- 失败数
- 未知数

### 9.2 面板结构

面板分为 5 个页签：

- `当前会话`
- `成功记录`
- `失败记录`
- `回放中心`
- `报告导出`

### 9.3 失败提示

失败接口实时 toast，内容只保留必要信息：

`创建订单失败: POST /api/order/create code=10023`

### 9.4 记录列表字段

每条接口展示：

- 方法
- URL
- HTTP 状态
- 业务码
- 判定结果
- 捕获时间
- 查看详情
- 生成模板 / 回放

### 9.5 详情抽屉

详情页展示：

- 请求头
- 请求体
- 响应头
- 响应体
- 判定依据
- 脱敏结果

## 11. 技术实现建议

### 11.1 捕获实现

#### `fetch`

- 缓存原始 `window.fetch`
- 包装入参
- 在响应返回后 clone 一份响应体做解析

#### `XMLHttpRequest`

- 重写 `open` 和 `send`
- 在 `readystatechange` 或 `loadend` 阶段采集结果

### 11.2 响应解析

按以下优先级解析：

- `application/json`
- `text/plain`
- 其他内容类型只记录摘要

### 11.3 脱敏策略

以下信息默认脱敏再存储或展示：

- `authorization`
- `cookie`
- `password`
- `mobile`
- `idCard`

### 11.4 性能控制

- 只记录命中规则的接口
- 响应体大小超过阈值时只截断保存
- UI 更新做节流
- 批量写入 IndexedDB

## 13. 风险清单

### 13.1 技术风险

- 某些请求签名带时间戳，回放必然失败
- 部分接口依赖前置接口返回值
- 某些站点对脚本改写 `fetch` 较敏感
- 页面跨域和 iframe 场景覆盖不全

### 13.2 业务风险

- 回放可能产生重复数据
- 测试账号状态可能污染
- 误把生产环境接口收录到模板中

### 13.3 缓解策略

- 默认测试环境白名单
- 高风险接口默认禁用回放
- 模板加环境标识
- 提供手工确认开关

## 15. 一周落地版本建议

如果目标是 1 周内做出第一版，建议砍到下面这个范围：

- 仅支持 `fetch` / `XHR`
- 仅支持 JSON 接口
- 仅支持单域名
- 仅支持单接口回放
- 仅支持基于 `cookie` / `localStorage` 的简单鉴权替换
- 报告只做基础统计和失败列表
